# Agent guide — codetalk

This file is written for AI coding agents (Claude, Codex, Cursor, etc.)
working inside a clone of this repo or inside a project that has copied
codetalk in.

## What codetalk does

Given a source file + markdown prose that references line ranges, it
produces an HTML page where scrolling the prose spotlights the matching
lines in a synced code pane. See `README.md` for the human-facing tour.

## Create a new codetalk (recipe)

Given: one or more source files to annotate, and a description of what
each annotated line range should explain.

1. **Drop source files under `_code/<slug>/`.** Example:
   `_code/auth-flow/login.py`. Pick a short kebab-case slug.
2. **Create `<slug>.md`** at the repo root with this front matter shape:

   ```yaml
   ---
   title: "Short descriptive title"
   scripts:
     - file: auth-flow/login.py
       label: login.py
     # one entry per file; `file:` is relative to _code/, `label:` is what
     # you use in ## headings below and what shows in the tab/header
   ---
   ```

3. **Write the body** using this exact structure — the transform in
   `codetalk.rb` is regex-based and brittle about heading shape:

   ```markdown
   ## login.py

   Optional preamble prose — appears above the codetalk grid as normal
   body text. One or more paragraphs. Keep under ~4 lines; longer
   preambles crowd the grid.

   ### Lines 7-15

   Annotation for lines 7–15. One idea per step. Two to four sentences
   works best.

   ### Lines 42-58

   Next step. Steps don't have to be contiguous; you can skip over
   uninteresting lines.
   ```

   Rules:
   - `## <label>` **must match** a `label:` from front matter exactly.
     Mismatches silently don't trigger the transform.
   - `### Lines N` or `### Lines N-M` — dashes or en-dashes both work.
   - Multiple `## <label>` sections back-to-back become tabs in one
     codetalk block. Insert any `#` or non-matching `##` between them to
     force a new block.
   - Two blank lines inside a step end the annotation; text after
     becomes normal body prose that rejoins the document flow.

4. **Build:**

   ```bash
   bundle exec ruby build.rb <slug>.md
   ```

   Writes `<slug>.html` beside the markdown. No arg → rebuilds
   `codetalk-1.md` → `index.html`.

5. **Verify** by opening `<slug>.html` in a browser (works on `file://`)
   or by scanning the output for the expected structure:

   ```bash
   grep -c '<div class="codetalk"' <slug>.html       # should be >= 1
   grep -c 'class="codetalk__step"' <slug>.html      # should equal the
                                                     # number of ### Lines
                                                     # headings
   ```

## Picking line ranges (for agents that can read source)

When a user gives you a source file and asks you to "walk through it" or
"explain it," your job is to pick the line ranges *and* write the prose.

- 3–7 steps per file is a good target. More than 10 and readers skim.
- Prefer ranges that correspond to a logical chunk — a class, a function,
  a config block, a branch of a conditional. Resist cutting through the
  middle of a construct.
- If the file is long, pick the "spine" — the lines that carry the main
  argument — and leave the rest un-annotated. The code pane truncates
  after the last annotated line, so unannotated tails are fine.
- Don't annotate imports unless they're load-bearing for the narrative.
- Each step's prose should explain *why* the code is shaped this way, not
  paraphrase what the code obviously does.

## Don't

- Don't invent `layout:` values; codetalk is layout-agnostic in Jekyll
  and ignored entirely by `build.rb`. Any `layout:` in front matter is
  passed through untouched — it doesn't affect codetalk.
- Don't reformat the source files under `_code/` to "clean them up." The
  line numbers in `### Lines N-M` are indices into the raw file — renames
  or reformats break every annotation.
- Don't add `codewalk__*` CSS classes in HTML overrides; the engine was
  renamed, only `codetalk__*` classes exist now.
- Don't inline source code in the markdown as fenced code blocks. The
  whole point of the split is that the source lives in `_code/` and is
  loaded + highlighted at build time.

## File roles

| File | Role | Touch it? |
|---|---|---|
| `codetalk.rb` | Transform + Jekyll plugin | Only to fix bugs in the engine |
| `codetalk.js` | Scroll spotlight client-side | Only to fix bugs |
| `_codetalk.scss` | Styles | Only for styling changes |
| `build.rb` | Standalone HTML builder | Usually no — takes CLI args |
| `<slug>.md` | Author writes these | Yes — this is the main output |
| `_code/<slug>/…` | Source files | Yes — copy the user's files in |
| `index.html` | Prebuilt sample output | Regenerated by `build.rb` |

## When in doubt

Read `codetalk-1.md` + `_code/codetalk-1/spaceship-region.py` together.
That pair is the canonical example of a well-shaped codetalk and should
answer most structural questions.
